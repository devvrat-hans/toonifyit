TOON（Token-Oriented Object Notation）は、LLMプロンプト専用に設計されたデータシリアライゼーション形式です。可読性と構造を維持しながら、JSONと比較してトークン使用量を30-60%削減します。TOON形式は配列にテーブル形式を使用し、オブジェクトには最小限のTOON構文を使用するため、JSON to TOON変換が非常に効率的です。

TOONは通常、JSONと比較してLLMトークンを30-60%節約します。大規模なデータセットやAPIレスポンスの場合、GPT-4やClaudeなどのAPIを使用する際に大幅なコスト削減につながります。正確な節約量はデータ構造によって異なります。

JSON vs TOON: TOONは、トークン効率のために設計されたよりコンパクトなTOON構文を使用します。TOONシリアライゼーション形式は、配列をヘッダー付きのテーブルとして表現し、最小限の句読点を使用し、冗長な文字を排除します。JSONは括弧、引用符、カンマでより冗長です。LLMプロンプト用にJSONをTOONに変換すると、TOONの効率性によりコストが大幅に削減されます。詳細なTOON vs JSON比較をお読みください。

はい！TOONは完全に可逆的です - データ損失なしでTOONをJSONに戻すことができます。当社の双方向コンバーターは、JSON to TOONとTOON to JSON変換の両方をサポートしています。TOONデータを貼り付けて「JSONに変換」を選択するだけで、元のJSON構造を取り戻すことができます。

TOONは、データベースクエリ結果、オブジェクトのリストを含むAPIレスポンス、分析データ、CSV形式のデータセット、eコマース製品カタログなどの均一なテーブルデータに最適です。複数のオブジェクトにわたって繰り返されるフィールドを持つデータは、最高のトークン節約（40-60%）を実現します。小規模なデータセットや深くネストされた構造では、改善が小さい場合があります。

はい、TOONはネストされたオブジェクトと配列をサポートしています。ネストされたオブジェクトはインデントベースの構造（YAMLのような）を使用し、ネストされた配列はその構造を維持します。ただし、TOONのトークン効率は、フラットで均一なデータで最も高くなります。複雑なネストされた構造でも機能しますが、劇的なトークン削減は見られない可能性があります。

TOONは、OpenAI（GPT-4、GPT-3.5）、Anthropic（Claude）、Google（Gemini）、Meta（LLaMA）などを含むすべての主要なLLMプロバイダーで動作します。TOONはテキストベースの形式であるため、テキストを処理できるLLMは、形式構造を説明する適切なプロンプト指示でTOONを理解できます。

TOONは3つの主要な最適化によってトークンを削減します：1) すべてのオブジェクトでフィールド名を繰り返す代わりに、配列ヘッダーで一度だけ宣言する、2) 最小限の句読点を使用し、不要な括弧/ブレースを排除する、3) 明示的な区切り文字の代わりにインデントベースの構造を採用する。これらを組み合わせることで、JSONと比較して30-60%のトークン削減を実現します。

TOONファイルは通常.toon拡張子を使用します（例：data.toon）。ただし、TOONはテキスト形式であるため、.txtファイルとして保存することもできます。コンバーターからダウンロードする場合、デフォルトの拡張子は.toonで、形式タイプを明確に識別します。

統合は簡単です：1) 言語用のTOONライブラリをインストール（JavaScript用はnpm install @toon-format/toon、Python用はpip install toon-format）、2) エンコード/デコード関数をインポート、3) LLMに送信する前にデータをTOONに変換、4) オプションでレスポンスをJSONに戻す。コード例とAPIリファレンスについては、ドキュメントをご確認ください。

はい、TOONはストリーミングLLMレスポンスで動作します。プロンプトの入力データにTOONを使用して、入力トークンを削減します。LLMのレスポンス（ストリーミングかどうかにかかわらず）は任意の形式にできます - TOONは主に入力トークンを最適化するように設計されています。LLMにTOONを出力させる必要がある場合は、プロンプトに形式指示を含めてください。

オンラインコンバーターは最大10MBのファイルを処理でき、ほとんどのユースケースに適しています。より大きなデータセットの場合は、コマンドラインツールを使用するか、TOONライブラリをアプリケーションに直接統合することをお勧めします。ライブラリ自体にはファイルサイズの制限はなく、非常に大きなデータセットを効率的に処理できます。
